# Real‑Time Dance Motion Capture and Robotic Execution with Reinforcement‑Learning Control

This report outlines the design and implementation strategy for a real‑time dance motion capture system that feeds a reinforcement‑learning (RL) policy to a humanoid robot.

* **Objective**: Capture a complete set of human skeletal keypoints, convert them to 3‑D coordinates, and use them to drive a robotic dance routine while preserving balance as the highest priority.
* **Key Innovations**: A comprehensive 55–60 key‑point model covering face, head, upper limbs, torso, lower limbs, and feet. Multi‑view (front, back, top) 3‑D reconstruction using a lightweight neural‑network backbone. A hierarchical RL architecture that interleaves a balance controller with a dance controller, switching automatically when the stability margin falls below a threshold.
* **Outcome**: A robust, low‑latency pipeline that can be deployed on an off‑the‑shelf humanoid platform, enabling expressive dance while guaranteeing safety.

## Key‑Point Coverage

| Body Part | Key‑Points (per side) | Total Points |
| --- | --- | --- |
| **Face** | 5 (eye corner, brow corner, mouth corner, nose tip, chin) | 5 |
| **Head** | 2 (pitch, yaw relative to horizontal plane) | 2 |
| **Upper Arm** | 3 per arm (shoulder, elbow, wrist) | 6 |
| **Hand** | 4 per finger $\times$ 5 fingers = 20 per hand | 40 |
| **Chest** | 3 (upper chest, mid‑chest, lower chest) | 3 |
| **Hips** | 2 (left/right hip center) | 2 |
| **Legs** | 2 per leg (hip, knee) | 4 |
| **Feet** | 5 per foot (toe base–toe tip) $\times$ 2 = 10 | 10 |
| **Total** |  | **66** |

> **Note:** The system aggregates the 66 points into 55–60 *effective* degrees of freedom (DOFs) by merging symmetric points and removing redundant joint angles (e.g., wrist orientation is derived from hand‑finger configuration).

## 3‑D Reconstruction Pipeline

### Multi‑View Data Acquisition

* **Front View**: RGB + depth or RGB‑only (if depth is unavailable).
* **Back View**: Similar sensor configuration.
* **Top View**: Overhead camera for foot‑plane reference.

All views are timestamp‑synchronized via a hardware trigger to guarantee temporal alignment.

### Pose Estimation Model

* **Backbone**: ResNet‑50 feature extractor followed by a transformer encoder for spatio‑temporal fusion.
* **Output**: 3‑D coordinates $pi \in \mathbb{R}^3$ for each key‑point $i$.

#### Equation of Position Estimation

$$pi = f_\theta(I_{\text{front}}, I_{\text{back}}, I_{\text{top}})$$

*(where $f_\theta$ denotes the neural network with parameters $\theta$.)*

### Coordinate Normalization

* The origin is set at the hip center.
* All coordinates are converted to the robot’s reference frame using a homogeneous transform $T_{\text{cam} \to \text{robot}}$.

### JSON Packaging

The JSON payload is transmitted over ROS 2 topics or ZeroMQ for real‑time ingestion by the RL policy.

```json
{  
  "timestamp": 1627841234.567,  
  "pose": {    
    "head": {"pitch": 0.12, "yaw": -0.04},    
    "face": {"eye_l": [0.01, 0.02, 0.09], "..." : "..."},    
    "shoulder_l": [0.03, 0.05, 0.10],    
    "toe_base_r": [0.02, -0.05, 0.00]  
  }
}
```

## Reinforcement‑Learning Control Architecture

### Hierarchical Control Loop

1. **High‑Level Planner**: Receives the full pose and computes a *desired gait* (balance) trajectory using an inverted‑pendulum model.
2. **Mid‑Level Dance Controller**: Receives a *dance intention vector* (e.g., “spin left”, “step forward”) and outputs joint torques.
3. **Low‑Level PID**: Maps torque commands to motor current references.

### State Representation

$$s_t = [p_t, \dot{p}_t, q_t, \dot{q}_t]$$

* $p_t$: 3‑D key‑point positions.
* $\dot{p}_t$: Estimated velocities via numerical differentiation.
* $q_t$: Joint angles (filtered).
* $\dot{q}_t$: Joint velocities.

### Reward Function

$$R = w_{\text{dance}}R_{\text{dance}} + w_{\text{balance}}R_{\text{balance}} - w_{\text{effort}}R_{\text{effort}}$$

* **Dance Reward**: Measures similarity to target choreography (DTW distance).
* **Balance Reward**: Penalizes deviation of the center‑of‑pressure (COP) from the support polygon:

$$R_{\text{balance}} = -\lambda \| \text{COP} - \text{COP}_{\text{support}} \|^2$$

* **Effort Reward**: Encourages energy efficiency.

### Prioritization Logic

| Condition | Action |
| :--- | :--- |
| $\| \text{COP} - \text{COP}_{\text{support}} \| > \epsilon_{\text{balance}}$ | Switch to balance mode; suppress dance policy. |
| $\| \text{COP} - \text{COP}_{\text{support}} \| \le \epsilon_{\text{balance}}$ | Execute dance policy. |

> $\epsilon_{\text{balance}} = 0.03\text{ m}$ (typical human stability margin).

The switch is realized via a *finite state machine* (FSM) implemented in the policy network’s hidden layer (policy gating).

## System Integration

### Data Flow

$$\text{Cameras} \to \text{Pose Estimator} \to \text{JSON} \to \text{ROS 2 topic} \to \text{RL Controller}$$

### Latency Budget

* **Pose Estimation:** 12 ms per frame.
* **Network & Parsing:** 2 ms.
* **RL Forward Pass:** 4 ms.
* **Total:** 18 ms ($\approx 55\text{ Hz}$).

### Hardware & Safety

* **Robot:** Heracles Humanoid Robot Platform V6.
* **Compute:** Raspberry Pi (Quad‑core ARM, 4 GB RAM).
* **Safety:** On‑board watchdog monitors balance margin. Emergency stop via a physical button or voice command.

## Evaluation Plan

| Metric | Method | Target |
| --- | --- | --- |
| **Pose Accuracy** | Mean per‑joint error vs. ground truth (motion capture studio) | $< 5\text{ mm}$ |
| **Balance Margin** | COP deviation | $< 0.03\text{ m}$ |
| **Dance Fidelity** | DTW score vs. choreography | $< 10\%$ |
| **Latency** | End‑to‑end profiling | $< 20\text{ ms}$ |
| **Robustness** | 10‑min continuous run with random perturbations | No falls |

## Conclusion

The proposed system integrates a fine‑grained 3‑D human pose estimator with a reinforcement‑learning controller that respects balance as the highest priority. By packaging the pose as a JSON payload and leveraging a hierarchical RL architecture, the robot can execute complex dance moves while maintaining safety. The outlined pipeline is ready for prototyping and experimental validation on a modern humanoid platform.
